# awei 阿魏怪诞正文配图

## 核心定位

为中文文章设计和生成 16:9 横版正文配图。目标不是制作商业插画、PPT 信息图、动漫海报或可爱吉祥物，而是把文章里的关键判断、流程、结构、状态与隐喻，转化成清爽、怪诞、有创意、可读但不过度说明的手绘解释图。

默认视觉 IP 是“阿魏”：黑色不规则刺猬头，头顶坐着黄色小鸭，半眯横眼、神情冷静，穿白色短袖，背后斜挂可变形的工具剑。阿魏是一名亲自进入系统、处理问题、推动结构运转的技术现场工作者。他必须承担画面的核心动作，不能只是站在旁边充当头像、贴纸或装饰。

## 先读这些参考

按任务需要读取，不要一次塞满上下文。文件名和路径保持不变：

- `references/style-dna.md`：风格 DNA、颜色、文字与禁忌。
- `references/xiaohei-ip.md`：阿魏 IP 的完整形象、性格、识别锚点、动作库和禁忌。文件名虽然保留 `xiaohei-ip.md`，内容与角色均以阿魏为准。
- `references/composition-patterns.md`：结构类型、原创隐喻方法和反复刻规则。
- `references/prompt-template.md`：单张生图提示词模板。
- `references/qa-checklist.md`：生成后的检查与迭代规则。
- `assets/examples/`：只用于低频视觉校准，不进入默认生成路径。不得照抄案例的构图、物件或标注。

开始设计或生成前，必须读取 `references/xiaohei-ip.md`。需要确定视觉密度和配色时再读取 `references/style-dna.md`；需要规划构图、编写提示词或验收结果时，分别读取对应参考文件。

## 阿魏的固定识别规则

每次出现阿魏时，应稳定保留以下视觉锚点：

- 大面积、略不规则的黑色刺猬头。
- 头顶或紧邻头顶的亮黄色小鸭。
- 半眯横眼与冷静、略带得意的微笑。
- 简单白色短袖、深灰长裤和白色运动鞋。
- 背后斜挂红棕色手柄的可变形工具剑。

小鸭用于远距离识别，也可以充当轻量的情绪指示器或系统状态灯。工具剑不是战斗武器，而是可变形的工程工具，可以切分信息、修复线路、撬动结构、测量边界、搭桥或撑开通道。

不得出现品牌标识或原头像中的 `HERO` 字样。不得照搬具体动漫角色、现成商业 IP 或特定画师风格。

## 工作流

### 1. 消化正文

先读取用户提供的正文、链接、Notion 页面、Markdown 文件或截图，提炼：

- 文章最重要的核心判断是什么。
- 哪些段落承担认知转折。
- 哪些内容适合通过图像解释。
- 哪些地方只适合保留文字，不需要配图。
- 阿魏能通过什么具体动作参与这个观点。

不要平均配图。优先寻找“认知锚点”，例如核心判断、关键断点、输入输出闭环、信息分流、前后对比、承接路径、常见风险和角色状态变化。

### 2. 把观点转换成动作隐喻

先用一句话概括每张图：

> 阿魏正在用什么动作，处理什么对象，表达什么观点。

动作必须具体，例如搬运、连接、切分、修复、筛选、组装、搭桥、撑门、检查、推动、拉回或扶正。避免使用“思考”“展示”“站在旁边看”等无法形成画面结构的弱动作。

如果去掉阿魏以后，核心隐喻仍然完整成立，说明角色只是装饰，需要重新设计动作。反过来，也不能为了让角色出现而破坏信息结构；当阿魏确实无法参与核心隐喻、只会增加噪声时，可以不出现。

### 3. 先出配图策略

如果用户只是要求“分析怎么配图”“思考哪些地方需要配图”或“给出配图建议”，先输出 shot list，不要直接生成。

每张候选图写清楚：

- 放在哪个段落之后。
- 图的主题与核心意思。
- 采用的结构类型。
- 阿魏正在执行的核心动作。
- 黄色小鸭和工具剑是否承担功能。
- 建议出现的关键元素。
- 建议使用的少量中文标注词。

默认规划 4–8 张；短文通常 1–3 张。长文也不要轻易超过 9 张。宁可减少数量，也不要把正文做成画册。

### 4. 单张生成

如果用户明确要求“生成”“输出”“做图”或“帮我生成”，不要停下来等待确认；使用内置 `image_gen`，每张图单独生成，不要把多张图拼成一张。

每张图只表达一个核心结构。提示词必须包含：

- 16:9 横版中文正文配图。
- 纯白纸张背景与大量留白。
- 粗而略不规则的黑色手绘线条。
- 扁平色块和极轻阴影，不使用写实光影。
- 少量红橙色与天空蓝中文手写批注。
- 阿魏的五个固定识别锚点。
- 阿魏作为核心动作主体，以及具体动作和被操作对象。
- 黄色小鸭或工具剑在本图中的明确作用。
- 禁止 PPT、商业插画、动漫海报、幼稚可爱、复杂架构和左上角类型标题。

提示词中先使用 `references/xiaohei-ip.md` 提供的固定角色描述，再追加当前画面的核心动作、被操作对象、结构关系、构图与少量标注。不得只写“画一个阿魏”，也不得依赖角色名字让生图模型自行猜测形象。

### 5. 保持原创构图

不得复刻过往案例。案例只用于校准留白、线条、配色密度和角色参与方式，不能直接复用“传送带断点”“角色拉线”“素材鱼”“盖章工具箱”“常见坑路径”等已有构图，除非用户明确要求复刻某一张图。

每次都应从当前文章重新发明一个奇怪但逻辑成立的隐喻。隐喻应让读者先看懂动作关系，再发现其中的荒诞感；不能只有怪异物件，却没有清晰观点。

### 6. 检查与迭代

生成后按 `references/qa-checklist.md` 检查，并额外确认：

- 阿魏的刺猬头、黄色小鸭、半眯眼、白色短袖和工具剑是否稳定。
- 阿魏是否正在执行一个可以用动词概括的关键动作。
- 小鸭或工具剑是否具有作用，而不是重复堆叠识别元素。
- 去掉人物后，核心隐喻是否会明显失去动作主体。
- 缩小图片后，黄色小鸭是否仍是清晰的视觉锚点。
- 同一组配图中的脸型、发型、服装、色彩和比例是否一致。

出现以下问题时，优先重生成或局部编辑：

- 阿魏只是摆姿势、指标题或站在角落围观。
- 人物特征漂移，黄色小鸭缺失，或工具剑变成战斗武器。
- 画面太满，缺少正文配图需要的留白。
- 太像流程图、PPT、商业插画或动漫海报。
- 中文过多、错字严重或依赖长句解释。
- 左上角出现“常见坑”“流程图”“系统架构图”等类型标题。
- 画风过度可爱、幼稚、圆润或闪亮。
- 背景不是干净白底。

### 7. 保存交付

如果用户在 workspace 内工作，将最终图片复制到：

```text
assets/<article-slug>-illustrations/
```

按顺序命名：

```text
01-topic-name.png
02-topic-name.png
```

保留原始生成文件，不要覆盖已有资产，除非用户明确要求替换。

## 输出口径

生成前的策略输出应短而准确。生成后的交付应包含：

- 一共生成了几张图。
- 每张图对应的正文位置和用途。
- 最终保存路径。
- 哪些图是核心必选，哪些图是可选补充。
- 如有明显一致性风险，指出需要继续修正的图片。

不要长篇解释风格理论，让图自己完成表达。
